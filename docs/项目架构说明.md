## 二、数据模型与关系

```
users (1) ──────────────── (N) thumbs_records
  │                              given_by → users.id（发放人）
  │                              admin_id → users.id（操作审计）
  │
  ├── (1) ──────────────── (N) exchange_records
  │                              product_id → products.id
  │
  ├── (1) ──────────────── (N) wishlists
  │
  ├── (1) ──────────────── (N) task_logs
  │                              task_id → tasks.id
  │
  └── (1) ──────────────── (N) tasks（created_by）
                                 reviewer_id → users.id（专属审批人）

products (1) ─────────────── (N) exchange_records
```

### 关键字段说明

**users.is_super_admin**

| 值 | 角色 | 权限 |
|----|------|------|
| `role=user` | 普通用户（飞行员） | 浏览、兑换、打卡、心愿 |
| `role=admin, is_super_admin=0` | 普通管理员（领航员） | 发放星辰币、管理自己的任务、审批自己负责的打卡 |
| `role=admin, is_super_admin=1` | 超级管理员（舰长） | 全部权限 |

**thumbs_records.thumb_type**

| 值 | 含义 | points 变化 |
|----|------|------------|
| `single` | 单星辰币 ⭐ | +1 |
| `double` | 双星辰币 🚀 | +5 |
| `deduction` | 阶梯扣除 | 由 `points` 字段决定（负整数） |

> 扣除时 `available_points = max(0, available_points + points)`，不产生负债。

**tasks.type**

| 值 | 含义 | 打卡限制 |
|----|------|---------|
| `daily` | 每日日常 | 同一用户同一任务当天仅可打卡1次 |
| `milestone` | 阶段里程碑 | 无次数限制 |

**tasks.reviewer_id**：`NULL` = 所有管理员均可审批；非 NULL = 仅指定管理员可审批。

**products.is_blind_box**：`1` 时为盲盒商品，兑换时后端从 `BLIND_BOX_PRIZES` 常量池随机抽取，结果写入 `exchange_records.remark`。

## 三、核心业务流程

### 星辰币发放（含阶梯扣除）

```
管理员选择用户
    ↓
选择模式：发放（单/双星辰币）or 扣除（选择档位或自定义）
    ↓
填写原因（必填）+ 舰长寄语（选填）
    ↓
提交 → 后端验证管理员权限
    ↓
扣除时：user.available_points = max(0, available_points + points)
    ↓
写入 thumbs_records（含 given_by、admin_id、parent_message）
更新 users.total_points 和 users.available_points
    ↓
前端流水：负数红色；≤-5 显示 ☄️ + 深红加粗；有寄语则展示提示条
```

### 任务打卡流程

```
用户在「任务大厅」选择任务点击「打卡」
    ↓
后端校验：
  - daily 类型：检查当天是否已存在 pending/approved 记录，有则拒绝
  - 任务是否 is_active
    ↓
写入 task_logs，status = 'pending'
    ↓
管理员在「任务管理 → 待审核打卡」中看到该记录
（普通管理员只看到 reviewer_id = 自己 或 NULL 的记录）
    ↓
批准 → 写入 thumbs_records（type='double' 或视奖励量）
       更新用户能量 → task_logs.status = 'approved'
驳回 → task_logs.status = 'rejected'
```

### 商品兑换（含盲盒）

```
用户浏览商城，盲盒商品显示金色卡片 + 🎁 标签
    ↓
点击兑换 → 弹出确认对话框（盲盒：展示"神秘奖品"提示）
    ↓
提交 → 后端校验：已上架 · 库存充足 · 能量足够
    ↓
若 is_blind_box == True：
    random.choice(BLIND_BOX_PRIZES) → 写入 remark
    ↓
写入 exchange_records
扣除 users.available_points
减少 products.stock
    ↓
返回 message（盲盒：含抽奖结果）→ 前端 ElMessage 展示
```

### 星际心愿单流程

```
用户提交心愿（title + expected_points）→ wishlists.status = 'pending'
    ↓
超级管理员在「心愿审核」查看待审核列表
    ↓
批准 → 自动在 products 写入新商品（下架状态）
        wishlist.status = 'approved'
拒绝 → wishlist.status = 'rejected'
```

## 四、技术选型

| 层级 | 选型 | 理由 |
|------|------|------|
| 前端框架 | Vue 3 + Vite | 组合式 API，构建速度快 |
| UI 库 | Element Plus | 组件丰富，与 Vue 3 原生契合 |
| 状态管理 | Pinia | Vue 3 官方推荐，轻量直观 |
| 图表 | ECharts | 功能完整，定制性强 |
| 后端框架 | Flask | 轻量灵活，适合中小型项目 |
| ORM | SQLAlchemy | Python 生态标准，防 SQL 注入 |
| 认证 | Flask-JWT-Extended | 无状态 Token，易于横向扩展 |
| 数据库 | MySQL 8.0 | InnoDB 事务支持，utf8mb4 字符集 |
| 容器化 | Docker + Nginx | 前端多阶段构建，生产部署标准化 |

## 六、安全设计

| 安全点 | 实现方式 |
|--------|---------|
| 密码存储 | Werkzeug `generate_password_hash`（scrypt） |
| 接口认证 | JWT Bearer Token，24 小时有效 |
| 权限校验 | 前端路由守卫 + 后端每个敏感接口单独校验 |
| 超级管理员校验 | 后端检查 `current_user.is_super_admin`，返回 403 |
| 任务权限隔离 | 普通管理员的任务/打卡 CRUD 均经过 `created_by`/`reviewer_id` 校验 |
| SQL 注入 | 全程使用 SQLAlchemy ORM，无裸 SQL 拼接 |
| XSS | Vue 模板自动转义，Element Plus 组件安全渲染 |
| CORS | Flask-CORS 全局启用，生产可收窄 origins |
| 文件上传 | 扩展名白名单 + 5MB 大小限制 + UUID 重命名防路径穿越 |
| 扣分保护 | `available_points = max(0, available_points + points)`，防止能量为负 |

## 七、扩展建议

- **积分排行榜**：基于 `users.total_points` 排序，新增一个 API 即可
- **操作日志**：新增 `operation_logs` 表，在关键操作写入记录
- **消息通知**：发放星辰币/打卡批准时向用户推送站内信
- **数据导出**：兑换/流水记录导出 Excel，后端引入 `openpyxl`
- **商品分类**：新增 `categories` 表，`products` 表加外键
- **积分过期**：`thumbs_records` 加 `expires_at` 字段，定时任务扫描清零
- **移动端适配**：现有 CSS 已有 `@media` 响应式基础，进一步完善即可
